# Getting a repo ready for chalkbag

This doc has two parts:

- **Repo onboarding** — run once per repo to make it agent-ready.
- **Dev onboarding** — run once per machine to keep outputs fresh automatically.

The target model is:

- repo instructions live in a tracked `AGENTS.md`
- Claude reads a tracked `CLAUDE.md -> AGENTS.md` symlink when Claude is enabled
- shared skills, provider config, and permission config live under `.chalk/`
- provider-native artifacts (`.claude/`, `.codex/`, `.opencode/`, `opencode.json`) are generated by chalkbag and gitignored

If you want an AI agent to do this work for you, point it at this file and tell it to follow the guide directly.

---

## Repo onboarding (run once per repo)

### 1. Scaffold the repo

```bash
cd ~/your-repo && chalkbag scaffold
```

This creates `.chalk/` from the built-in template and an `AGENTS.md` stub if they are missing. It is idempotent — running it again will not overwrite your edits.

To also register the repo with the daemon and run the first build in one step:

```bash
cd ~/your-repo && chalkbag init
```

`chalkbag init` scaffolds, registers the repo in `repo` mode, and performs a synchronous first build. It prints the providers rendered and the paths written.

### 2. Edit the tracked root `AGENTS.md`

Replace the placeholder repo description, repo map, and working rules. Keep the file repo-specific. This file is your source of truth for agent instructions — not `.chalk/`.

Write it as a **map, not a README**: a one-line repo purpose, a directory map (`path -> what lives there -> when to read it`), pointers to the real docs, the exact build/test/lint commands, and the working rules an agent cannot infer from the code. Do not paste in architecture essays or API reference — link them. Aim for ~60-120 lines. The full doctrine, with good/bad examples, is in [authoring-agents-md.md](./authoring-agents-md.md).

### 3. Edit `.chalk/README.md`

Put chalkbag workflow and tooling notes here, not in root `AGENTS.md`.

### 4. Review `.chalk/providers.yaml`

This file is required for schema validation. It declares which providers are active for this repo:

```yaml
providers:
  claude:
    enabled: true
  codex:
    enabled: true
  opencode:
    enabled: true
```

The `enabled` field controls schema validation but not which providers `chalkbag build` renders — pass `--provider <ids>` to the build command to restrict rendering.

### 5. Import shared skills (optional)

Add a `.chalk/config.yaml` to reference external sources:

```yaml
imports:
  - source: github:<org>/<repo>
    ref: v1.2.0
    path: skills/oncall
```

Local files in `.chalk/` always win over imported files. The `ref` can be a branch, tag, or commit SHA. See [agents-spec.md — External imports](./agents-spec.md#6-external-imports) for details.

### 6. Adjust optional inputs if needed

- Edit `.chalk/permissions.yaml` if the repo needs provider-native permission rendering.
- Add shared skills under `.chalk/skills/<skill-name>/`.

### 7. Extend the repo's `.gitignore`

Append this snippet to your repo's `.gitignore`:

```gitignore
# chalkbag — generated provider-native artifacts
.claude/
.codex/
.opencode/
opencode.json
```

Do not ignore the tracked root `AGENTS.md` or the tracked `CLAUDE.md` symlink.

### 8. Render once

```bash
cd ~/your-repo && chalkbag validate && chalkbag build --yes
```

`validate` checks the merged source tree against the schema, including any resolved imports.
`build` writes the generated provider-native outputs and updates `.chalk/.state.json`.

Use `--provider <ids>` only as a one-off override:

```bash
cd ~/your-repo && chalkbag build --provider claude,codex --yes
```

If you omit `--provider`, chalkbag falls back to the last rendered provider set, or renders all providers if there is no prior state.

### 9. Add scoped guidance and the Claude bridge symlinks

First, create the root Claude bridge if Claude is enabled:

```bash
cd ~/your-repo && ln -sf AGENTS.md CLAUDE.md
```

Claude reads `CLAUDE.md`; Codex and the rest of the agent ecosystem read `AGENTS.md` directly. Commit this symlink.

Next, add scoped `AGENTS.md` files in self-contained domains — `packages/*`, `services/*`, or major `src/` areas — so local conventions, gotchas, and commands live next to the code they describe instead of bloating the root. Keep each scoped file local-only (never repeat the root), and list every scoped guide in a table in the root `AGENTS.md`. Placement and content rules are in [authoring-agents-md.md](./authoring-agents-md.md).

Every scoped `AGENTS.md` that Claude should see needs its own sibling symlink:

```bash
cd ~/your-repo/packages/server && ln -sf AGENTS.md CLAUDE.md
```

### 10. Verify the result

```bash
cd ~/your-repo && chalkbag doctor
```

`chalkbag doctor` reports the daemon status, heartbeat freshness, config paths, and whether the rendered outputs are up to date.

## Dev onboarding (run once per machine)

### Install the daemon

`chalkbag daemon install` detects your platform and installs the right service manager: a launchd user agent on macOS, a systemd user unit on Linux. The command sequence is identical on both:

```bash
chalkbag daemon install
chalkbag daemon status
```

`chalkbag daemon status` reports the service-manager liveness (`launchctl` on macOS, `systemctl --user is-active` on Linux) merged with the heartbeat freshness check, so it reads coherently on either platform.

#### macOS (launchd)

The unit is a plist at `~/Library/LaunchAgents/com.chalkbag.daemon.plist`, started via `launchctl bootstrap`.

#### Linux (systemd user unit)

The unit is written to `~/.config/systemd/user/chalkbag.service` (respecting `XDG_CONFIG_HOME`) and activated with `systemctl --user enable --now`. Logs go to the journal:

```bash
journalctl --user -u chalkbag.service -f
```

On a headless box the systemd **user** session is torn down at logout, which stops the daemon. Enable lingering so it keeps running (and starts at boot):

```bash
loginctl enable-linger "$USER"
```

If `chalkbag daemon install` reports that the user session bus is unavailable, that lingering step (or logging into a graphical/SSH session with `XDG_RUNTIME_DIR` set) is what it needs. You can always fall back to the foreground `chalkbag watch` (below) with no daemon at all.

The daemon watches all registered paths and rebuilds on file change. After installing, register your repos:

```bash
chalkbag register ~/your-repo
```

To register a parent directory that contains many repos (chalkbag watches one level deep for repos with `.chalk/`):

```bash
chalkbag register --parent ~/Documents/Projects
```

Or use the alias:

```bash
chalkbag register-group ~/Documents/Projects
```

Check what is registered:

```bash
chalkbag paths
```

### No-daemon fallback / inline watcher

If you prefer not to install the daemon on any platform — or you are on a headless Linux box where enabling a user session is not worth it:

```bash
cd ~/your-repo && chalkbag watch
```

`chalkbag watch` is an inline watcher that stays in the foreground. It is equivalent to the daemon's per-repo watcher but exits when you close the terminal. It is also the fallback on platforms without a supported service manager (anything other than macOS or Linux).

## Machine-level onboarding (run once per machine)

Global scope applies the chalkbag model to your home directory: a `~/.chalk/` source tree compiled into your **user-level** Claude and Codex config. It is a machine-wide baseline; any repo's own `.chalk/` still overrides it inside that repo. For the full contract, see [agents-spec.md — Global scope](./agents-spec.md#10-global-scope-machine-level-chalk).

### 1. Scaffold and build the global scope

```bash
chalkbag init --global
```

This scaffolds `~/.chalk/` (with `providers.yaml`, `skills/`, and a machine-level `AGENTS.md`), registers the global scope with the daemon, and runs the first build. It is idempotent.

### 2. What gets linked and written

- `~/.claude/CLAUDE.md -> ~/.chalk/AGENTS.md` and `~/.codex/AGENTS.md -> ~/.chalk/AGENTS.md` (bridge symlinks).
- Skills under `~/.chalk/skills/` are projected into `~/.claude/skills/` and `~/.agents/skills/` (the Codex / AGENTS.md-spec user-level skill dir).
- If you add a `~/.chalk/permissions.yaml`, it merges into `~/.claude/settings.json` (your entries are never dropped) and a managed block inside `~/.codex/config.toml`.

If `~/.claude/CLAUDE.md` or `~/.codex/AGENTS.md` already exists as a regular file with content, the build fails and asks you to merge that content into `~/.chalk/AGENTS.md` and re-run — it never clobbers your existing instructions.

### 3. Edit and rebuild

```bash
# edit machine-wide guidance / skills / permissions
vim ~/.chalk/AGENTS.md

chalkbag validate --global
chalkbag build --global
```

### 4. Daemon pickup

`chalkbag init --global` registers a `global` entry in the daemon registry. Once the daemon is installed (`chalkbag daemon install`), it watches `~/.chalk/` and rebuilds the global scope on every change, exactly like it watches your repos. Check it with:

```bash
chalkbag doctor   # the `global` block shows the source path, whether it is registered, and its providers
chalkbag paths    # the global entry appears with "mode": "global"
```

### 5. Undo

```bash
chalkbag clean --global
```

removes only chalkbag-written outputs (skill projections, the codex managed block, and the bridge symlinks when they still point at `~/.chalk/AGENTS.md`). It never deletes `~/.claude/settings.json` or any content outside the managed block.

> **Warning — revoking a global permission requires a manual edit.** Global permissions are *unioned* into `~/.claude/settings.json`, so chalkbag only ever adds rules there, never removes them. Removing a rule from `~/.chalk/permissions.yaml` does **not** take it back out of `~/.claude/settings.json`, and `chalkbag clean --global` does not either. To actually revoke a global rule, delete the entry from `~/.claude/settings.json` yourself.

## Day-to-day workflow

- Edit `.chalk/` files and root `AGENTS.md`. Never hand-edit `.claude/`, `.codex/`, `.opencode/`, or `opencode.json` — the next render overwrites them.
- Keep `CLAUDE.md` as a symlink. If it becomes a regular file, recreate it:

  ```bash
  cd ~/your-repo && ln -sf AGENTS.md CLAUDE.md
  ```

- Run `chalkbag validate` before `chalkbag build --yes` when making structural changes to `.chalk/`.
- The daemon or `chalkbag watch` keeps outputs fresh. Git hooks are the fallback: if the daemon heartbeat is stale, commits still trigger an inline rebuild.
- To temporarily pause daemon rebuilds for a repo (for example, during a long refactor):

  ```bash
  chalkbag daemon pause
  ```

  Resume when ready:

  ```bash
  chalkbag daemon resume
  ```

## Troubleshooting

See [troubleshooting.md](./troubleshooting.md) for a full list of symptoms and fixes.

Quick reference:

| Symptom | Fix |
|---|---|
| `chalkbag doctor` reports artifacts not rendered | `chalkbag build --yes` |
| `chalkbag doctor` reports daemon not installed | `chalkbag daemon install` (launchd on macOS, systemd on Linux), or use `chalkbag watch` |
| Generated directories keep reappearing in `git status` | Append the `.gitignore` snippet from step 7 above |
| `CLAUDE.md` was committed as a regular file | `ln -sf AGENTS.md CLAUDE.md` and commit the symlink |
| `.chalk/` edits do not propagate | `chalkbag daemon status`; reload or resume if needed |
